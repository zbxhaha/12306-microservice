# 12306 铁路购票微服务项目

12306 铁路购票服务是与生活和出行高度相关的分布式高并发系统，包含会员、购票、订单、支付和网关等核心服务。

本项目基于开源项目学习改造，用于深入理解分布式系统设计，重点掌握高并发、缓存、分库分表、消息队列、设计模式等后端核心知识点，非常适合作为校招简历项目进行学习与展示。

项目提供两种独立运行版本：

- **SpringBoot 聚合服务版本**：适合测试和部署，可以直接启动 aggregation-service 聚合服务和网关服务。
- **SpringCloud 微服务版本**：适合学习微服务设计，可以分别启动支付、订单、用户、购票和网关服务。

可根据学习和使用需求选择对应版本启动，微服务版本侧重架构设计学习，聚合服务版本侧重测试与部署验证。

## 项目质量与设计难点

以用户服务系统为例，高并发海量数据下需要考虑很多设计要点：

1. 用户注册、添加乘车人时，如何校验信息真实性，保障系统安全？
2. 海量用户数据如何分库分表，选择什么分片键，如何平滑上线与回滚？
3. 多方式登录（用户名、手机号、邮箱）导致的“读请求扩散”如何解决？
4. 高并发注册场景下，如何有效解决缓存穿透问题？
5. 姓名、手机号、证件号等敏感信息，如何防止泄露？

以购票服务为例，当用户购买两个乘车人的高铁一等座票且没有选座时，座位分配逻辑如下：

1. 检查列车一等座余票是否足够，不足则直接返回购票失败；
2. 获取有两个座位余票的车厢并遍历处理；
3. 优先查找同车厢相邻座位；
4. 无相邻则查找同车厢不相邻座位；
5. 仍不满足则分配不同车厢不相邻座位；
6. 在余票充足前提下，保证购票成功并尽可能优化座位体验。

## 12306 购票业务误区

背景：列车途径北京南、济南西、南京南、杭州东、宁波。

### 车次查询
- 误区：使用 ElasticSearch 实现
- 实际：高并发下 ES 不适用，采用 Redis 缓存 + 内存组装实现。

### 购票扣票逻辑（北京南 → 南京南）
- 误区：只扣当前区间票
- 实际：同时扣减北京南-济南西、北京南-南京南、济南西-南京南，任一不足则购票失败。

### 中间站购票（济南西 → 南京南）
- 误区：刷中间站会导致大站无票
- 实际：通过动态放票解决，初期优先大站票量，临近发车再放开小站票。
